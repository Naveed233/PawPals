PawPair · Phase 1 backend

Architecture & data model

A static Expo app talks straight to Supabase over HTTPS — no app server to run. Postgres owns the truth, Row Level Security owns who sees it, and two database triggers do the only work clients can't be trusted with.

Expo / React Native GitHub Pages (static) Supabase Auth Postgres + RLS Storage Realtime (phase 2)

System

How the pieces connect

One client bundle serves web, iOS, and Android. The only credentials in it are the publishable key — safe by design, because every table is gated by RLS policies, not by the client.

Data model

Tables & relationships

profiles extends Supabase's built-in auth.users one-to-one. Everything else hangs off profiles. Dog ids are text so demo dogs (dog-luna) and real dogs (UUIDs) can coexist in swipes.

matches stores the pair ordered (owner_a < owner_b) with a unique constraint — one match per pair, no duplicates regardless of who liked last.

Triggers

The two things clients can't be trusted with

Both run inside Postgres with definer rights. The app never inserts into profiles at signup or matches at all — it can only write its own swipes.

Security

Row Level Security matrix

Policies apply to signed-in users; anonymous requests see nothing at all. This is the entire authorization layer — there is no API server to bypass.

TableReadCreateUpdateDelete
profiles all signed-in own row own row —
dogs all signed-in own dogs own dogs own dogs
swipes own swipes own swipes — own swipes
matches participants trigger only — —
messages participants participants — —
events all signed-in as host — —
rsvps all signed-in own RSVP — own RSVP
storage / photos public URL own folder (uid/…) — —
own scoped to auth.uid() all signed-in discovery-grade data participants both sides of a match — no policy → denied

Profile visibility to matches (the in-app privacy toggle) is stored in profiles.show_profile_to_matches; enforcing it at the policy level is a phase-2 refinement along with Realtime chat and Storage uploads.
